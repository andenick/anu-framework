## 1. Purpose

`anu-rebuild` wraps the salvage-and-port workflow for taking an existing data construction project (typically a predecessor like ST2 or an academic replication that lives outside the Anu Framework) and producing a fresh, Anu-Framework-native implementation.

It is **the meta-skill that orchestrates the other 19 skills** through a 6-wave + closeout cadence proven to take a 60-series replication from blank scaffold to 100% PASS distribution-ready in **~12-15 focused sessions** (reference-replication baseline). With the friction-point remediations in [`ANU_FRAMEWORK_IMPROVEMENTS_RFC.md`](ANU_FRAMEWORK_IMPROVEMENTS_RFC.md) absorbed, the target drops to **~7-10 sessions**.

The skill is general: it works on any large data-construction predecessor regardless of domain. The reference-replication rebuild was economic data; the same workflow applies to capital-markets, sectoral-employment, environmental-accounts, or any other domain where you have a predecessor project + book/source documents + the intention to reauthor under audit-grade discipline.

---

## 2. When to invoke

Invoke `anu-rebuild` when **all four** of these conditions hold:

1. **A predecessor project exists** with valuable data, methodology, and/or correspondence that should be salvaged rather than recreated from raw sources.
2. **The target is a new public-facing repository**, fully Anu-Framework-native, ready for `/anu-publish` distribution.
3. **Series count is large enough** (≥20) that ad-hoc rewriting would lose discipline. Below 20 series, use `anu-pipeline` directly — the rebuild overhead exceeds its benefit.
4. **The predecessor is NOT already Anu-Framework-native at the current version.** If it is, you only need `anu-ingestion migrate-scheme` (Friction 1) — not the full rebuild workflow.

### Examples that DO fit

- ST2 (T###/N#### naming, hardcoded internal paths, multi-phase pipeline but no Anu skill alignment) → the reference replication
- A heterodox-economics replication that lives in a Jupyter notebook → fresh `Technical/` project
- An academic data package with no per-series provenance → series_registry.json + 64 DPRs

### Examples that do NOT fit

- A project with no predecessor → use `anu-pipeline init` to scaffold fresh
- A 5-series side project → too small for rebuild overhead; per-series with existing skills
- A predecessor already at Anu Framework v10.0 needing v11.0 bumps → use `anu-ingestion migrate-scheme` only

## 3. Inputs

| Input | Type | Description |
|---|---|---|
| `predecessor_root` | path | Filesystem path to the prior project. Will be read-only during rebuild. |
| `target_root` | path | Filesystem path for the new build. `anu-rebuild scaffold` creates the tree here. |
| `prefix_scheme` | JSON | Declaration of the new ID prefix scheme. Example: `{"primary":"S","external":"ES","analytical":"AS"}`. |
| `salvage_manifest` | CSV (optional) | Explicit list of predecessor artifacts to copy into `Inputs/Salvaged/`. If absent, `anu-rebuild salvage` proposes a default list and asks for approval. |
| `crosswalk_table` | CSV | Mapping `predecessor_id,new_id` (and optionally `predecessor_subseries_id,new_subseries_id`). Generated by `anu-rebuild crosswalk` from registry inspection; agent reviews and approves. |
| `wave_plan` | YAML (optional) | Per-wave cohort assignments. Example: Wave 1 = chapter 5 series, Wave 2 = chapter 6, etc. If absent, agent prompts for the partition. |

## 4. Six-wave workflow

Each wave is a discrete sub-command. The workflow is sequential but waves can sub-batch (sprints).

### Wave 0 — Foundation

**Goal**: empty scaffold + curated salvage + registry skeleton + crosswalk + framework audit clean.

Sub-commands:

```bash
anu-rebuild salvage <predecessor_root> --to Inputs/Salvaged/ [--manifest salvage_manifest.csv]
anu-rebuild crosswalk --predecessor Inputs/<pred>/series_registry.json \
                      --prefix-scheme '{"primary":"S","external":"ES","analytical":"AS"}' \
                      --output MIGRATION/crosswalk.csv
anu-rebuild scaffold --target . --crosswalk MIGRATION/crosswalk.csv [--from-template reference-replication]
anu-doctor                        # framework consistency check (existing skill)
```

What each does:

| Sub-command | What it produces |
|---|---|
| `salvage` | `Inputs/Salvaged/` populated with the predecessor's irreplaceable artifacts: book text extractions, correspondence, methodology decisions, canonical CSVs. Read-only afterwards. |
| `crosswalk` | `MIGRATION/crosswalk.csv` with `old_id,new_id,name,chapter_or_source,year_range,notes`. One row per predecessor series. |
| `scaffold` | The full `Technical/` tree: `series_registry.json` skeleton, empty pipeline phases, `.gitignore`, `.publish_ignore`, `data/{source,raw,intermediate,final}/`, `code/{S00_setup,L01_loaders,P02_processors,V03_validators,M04_manual,A05_analysis,O06_output,utils}/`, `docs/{series,chapters,methodology,decisions}/`. Initialized git repo with initial commit. |

**Acceptance** for Wave 0:

- `Inputs/Salvaged/` has the curated artifacts; `Inputs/<predecessor>/` is untouched
- `MIGRATION/crosswalk.csv` exists and is reviewed/approved
- `series_registry.json` has the prefix_scheme block and all series stubs from crosswalk
- `anu-doctor` clean (existing skill)
- First git commit: `"Wave 0: foundation - scaffold + salvage + crosswalk"`

---

### Wave 1..N — Per-cohort implementation

**Goal**: implement series in cohort batches (chapter, wave, study group). Each cohort cycles through the Anu pipeline.

For each cohort:

```bash
anu-rebuild wave-execute --cohort <name> [--strict]
```

Internally `wave-execute` orchestrates:

1. **Research** — `anu-research mine-chapter <chapter>` OR `anu-research port --from <predecessor>/research/ --id-map MIGRATION/crosswalk.csv` (Friction 6)
2. **Adequacy gate** — `anu-adequacy <chapter>` (must score ≥80 ADEQUATE)
3. **Ingestion** — `anu-ingestion batch-create-dpr --cohort <name> --template default` (Friction 2)
4. **Extension provenance** — `anu-extension batch-create-epr --cohort <name>` for the extendable subset
5. **Code scaffolding** — `anu-scaffold generate --cohort <name>` (Friction 8). Auto-selects template per series's `content_type` and `construction`.
6. **Agent fills scaffold stubs** with construction-specific logic. Loaders pull from `data/source/` or `data/raw/`; processors implement the registry's `construction` steps; validators encode benchmarks from the registry's `validation` block.
7. **Run pipeline** — `python run.py --series <each in cohort> --validate-only` until all PASS
8. **Output generation** — `anu-chopped --cohort <name>` and `anu-extenbook --cohort <name>`
9. **Review gate** — `anu-review <chapter>` must score ≥85% scope-adjusted with D13 (data authenticity) and D14 (outward intelligibility) GREEN
10. **Cohort commit** — `git commit -m "Wave <N>: <cohort> closeout - N series PASS"`

**Acceptance** per cohort:

- Every series in the cohort has status `validated:<scope>` (per Friction 7 taxonomy)
- `anu-review <chapter>` ≥85%, D13 GREEN
- All artifact types present per series: research JSON, DPR, decomposition, L01 (or skip if derived), P02, V03, chopped CSV, extenbook xlsx
- Cohort closeout commit
- `anu-doctor project` (Friction 12) CLEAN for the cohort scope

---

### Wave N+1 — Distribution

**Goal**: produce the three public distribution artifacts and push.

Sub-commands:

```bash
anu-publish audit                              # Friction 4 — iterate until CLEAN
anu-publish generate --profile data+pipeline   # public GitHub repo tree
anu-drive generate                              # Google Drive consumer bundle
anu-archive generate                            # audit-grade .zip
anu-review --whole-project                     # final review; D13 + D14 must be GREEN
git remote add origin <public_repo_url>
git push -u origin main
gh release create v1.0 Outputs/Archive/*.zip
```

**Acceptance** for Distribution wave:

- `anu-publish audit` reports CLEAN (zero internal references)
- `Outputs/Publish/` is a valid GitHub-ready tree
- `Outputs/Drive/` opens without code; contains master xlsx + per-series workbooks + methodology PDF
- `Outputs/Archive/...zip` extracts; SHA-256 manifest verifies
- Fresh-clone smoke test: `git clone`, `pip install`, `python run.py --validate-only` completes in <5s
- `anu-review` whole-project ≥85% with D14 GREEN (required for external distribution)
- GitHub Release v1.0 published with archive attached
- Drive folder shared publicly; link added to GitHub README

---

### Wave N+2 — Polish

**Goal**: project-level audit, KB enrichment, skill-update RFCs.

Sub-commands:

```bash
anu-doctor project                              # Friction 12
anu-doctor project --check P01_dpr_coverage --strict
/sphdarp <chapter_pages>                        # KB enrichment if D1 < 80 anywhere
```

If `anu-doctor project` flags issues:
- P01-P10 failures → fix and re-run before declaring done
- P05 stale references → these are bugs; resolve them

If the build surfaced framework gaps (it always does; the build is itself a test of the framework):
- Log them in `MIGRATION/skill_updates_RFC.md`
- These become next-cycle framework PRs

**Acceptance** for Polish wave:

- `anu-doctor project` CLEAN
- KB coverage ≥80% across all chapters
- Framework friction points logged in `MIGRATION/skill_updates_RFC.md`
- Final commit closing out the rebuild

## 5. Cross-cutting policies

These are invariants enforced across all waves. Violations should be detected by `anu-doctor project`.

### 5.1 `Inputs/Salvaged/` is read-only

Once Wave 0's `salvage` sub-command completes, **nothing else writes to `Inputs/Salvaged/`**. Pipeline code does not reference the predecessor's `Inputs/<predecessor>/` tree directly. All curated copies live in `Inputs/Salvaged/`. The salvage list is auditable.

### 5.2 Crosswalk maintained for one release cycle

`MIGRATION/crosswalk.csv` is the canonical mapping between predecessor IDs and new IDs. It is referenced during research-JSON port, ID-scheme migration, and consistency checks. After the first public release (v1.0), the crosswalk is frozen as a historical doc; subsequent updates use the new IDs natively.

### 5.3 Divergences logged centrally

Every divergence from the predecessor — whether ingestion-phase, extension-phase, or manual adjustment — is logged via `register_divergence()` (Friction 3) into the top-level `DIVERGENCE_REGISTER.json`. No silent rewrites. The reference-replication build surfaced one (S507 surplus ratio NIPA proxy → book-faithful e/(1+e)); a more conventional rebuild might surface zero or many. The point is none are hidden.

### 5.4 Status field discipline

Every registry entry declares a status from the standardized enum (Friction 7). Pending series carry explicit dependency tokens (`pending:K_star`, `pending:NIPA_loader`). Partial coverage carries explicit reason (`partial:14_year_coverage_only`). `anu-doctor project` validates.

### 5.5 No synthetic data, ever

`anu-ingestion`'s Source Authenticity Verification (Section 8 of v4.0) applies universally. Pending series carry explicit `NaN` with provenance markers, not interpolated guesses. The framework's whole authenticity discipline (D13 gate in `anu-review`) is upheld throughout the rebuild.

### 5.6 Per-wave closeout commits

Each wave ends with a git commit whose message lists every series gained, every artifact produced, and the validation status. The git log is the project progress report. No silent merges.

## 6. Anti-patterns

Things `anu-rebuild` explicitly prohibits or warns against, all observed in predecessor projects:

### 6.1 Do not copy predecessor outputs as starting point

The predecessor's `data/final/*.csv` are reference benchmarks at most. Re-derive every value from the documented source (book table, BEA API cache, external study CSV). Copying outputs creates silent dependencies; if the predecessor had a bug, your rebuild inherits it.

**Real example**: ST2's `T507.csv` carried the NIPA-proxy surplus ratio with values like 0.5698 at 1948. The book itself implies S/Y = e/(1+e) = 0.6296 (algebraic identity). The reference replication's S507 was re-derived from S505 and S504; the divergence was caught and documented. Had I copied ST2's T507.csv, the bug would have shipped.

### 6.2 Do not skip per-wave review

The `anu-review` gate is the framework's quality enforcement. Skipping pushes defects downstream where they are 10x more expensive to fix. The 85% gate is meant to be a hard line; if a cohort can't clear it, fix the cohort before starting the next one.

### 6.3 Do not fabricate "pending" data

Series that can't be implemented yet (missing K\*, missing NIPA loader, etc.) carry explicit `NaN` values in their final CSVs and `status: pending:<dependency>` in the registry. NEVER fill pending years with synthetic averages, linear interpolations, or "reasonable estimates." `anu-doctor project` checks for `np.random` calls and synthetic markers.

### 6.4 Do not let the crosswalk drift

Once `MIGRATION/crosswalk.csv` is approved at Wave 0, every artifact uses the new IDs consistently. No file should reference the predecessor IDs except (a) the crosswalk itself, (b) salvaged read-only artifacts in `Inputs/Salvaged/`, (c) the `MIGRATION/_port_*.py` one-off porters.

### 6.5 Do not write generators outside `MIGRATION/`

One-off scripts that don't belong in the public release live in `MIGRATION/`. The directory is in `.publish_ignore` by default. Keeps the public tree clean. The 17 reference-replication generators were all in `MIGRATION/`.

## 7. When NOT to use `anu-rebuild`

| Scenario | Use instead |
|---|---|
| No predecessor; fresh project | `anu-pipeline init` directly |
| Predecessor already at current Anu Framework version | `anu-ingestion migrate-scheme` only (Friction 1) |
| Series count <20 | Per-series invocation of existing skills; rebuild overhead exceeds benefit |
| Predecessor's data is unsalvageable (corrupted, lost) | Treat as fresh project with extra digitization step |
| Single-chapter rebuild within a larger Anu-native project | `anu-pipeline --from-stage 3` for that chapter; not the full rebuild workflow |

## 9. Integration with other Anu skills

| Skill | Relationship |
|---|---|
| `anu-doctor` | Required dependency. Runs at end of Wave 0 (framework audit) and Wave N+2 (project audit) |
| `anu-ingestion` | Heavy consumer. Wave 0 uses `migrate-scheme`; every per-cohort wave uses `batch-create-dpr` |
| `anu-research` | Per-cohort wave step #1. Optionally `port` from predecessor |
| `anu-adequacy` | Per-cohort wave step #2 (gate) |
| `anu-extension` | Per-cohort wave step #4 (extendable subset) |
| `anu-scaffold` | Per-cohort wave step #5 (Friction 8 new skill) |
| `anu-chopped`, `anu-extenbook` | Per-cohort wave step #8 |
| `anu-review` | Per-cohort wave step #9; Wave N+1 whole-project audit |
| `anu-publish`, `anu-drive`, `anu-archive` | Wave N+1 distribution |
| `anu-pipeline` | `anu-rebuild` overlays a project-scoped workflow on top of `anu-pipeline`'s 8-stage skill orchestration |

The skill does NOT replace `anu-pipeline` — it sits *above* it, prescribing the project-scope cadence while `anu-pipeline` handles within-stage orchestration.

## 10. Generalization beyond economic data

The reference-replication build was economic data (Shaikh-Tonak Marxian aggregates). The workflow generalizes to any domain that meets the four "when to invoke" conditions in §2.

**Examples of how the same workflow maps to other domains**:

| Domain | Predecessor | Salvage candidates | Per-cohort partition |
|---|---|---|---|
| Capital markets | Bloomberg ETL pipeline + analyst spreadsheets | Cleaned tick data, methodology notes, correspondence with quants | by asset class (equities, fixed income, FX, ...) |
| Sectoral employment | BLS CES historical Jupyter notebook | Bridge tables, classification mappings | by NAICS sector |
| Environmental accounts | EPA + Eurostat compiled datasets | Concordance tables, satellite-data documentation | by pollutant / by country |
| Heterodox replication (Mohun, Foley, Duménil-Lévy) | Author's R or Stata code | Author correspondence, alternative-classification spreadsheets | by chapter / by paper |

In every case the 6-wave cadence applies. Only the per-cohort partition varies. The salvage targets, crosswalk semantics, and scrub patterns are domain-agnostic.

## 11. Verification — how to know `anu-rebuild` is working

The skill is correctly implemented and useful when:

1. **A new rebuild target measured against reference-replication baseline shows ≥30% session-count reduction** (12-15 → 7-10 sessions). This is the primary metric.
2. **The 6-wave cadence produces complete artifacts without ad-hoc generator scripts.** The 17 reference-replication generators all become unnecessary; their work absorbed by `anu-scaffold`, `anu-ingestion batch-create-dpr`, and `anu-rebuild salvage/crosswalk`.
3. **Every divergence from the predecessor is logged in `DIVERGENCE_REGISTER.json`** — not in ad-hoc markdown.
4. **`anu-doctor project` is CLEAN at Wave N+2.**
5. **The public release passes the scrub audit and external review** without manual scrubbing iterations.
6. **The 14 reference-replication empirical findings (or domain-equivalent findings) are reproduced** post-rebuild.

## 13. Open questions for maintainer review

1. **Skill name**: `anu-rebuild` (chosen, emphasizes from-scratch rewrite) vs. `anu-port` (emphasizes data migration). Recommend `anu-rebuild` since the workflow re-authors *code*, not just data.
2. **Should `anu-rebuild` invoke `anu-scaffold` directly, or delegate?** Recommend direct invocation: `anu-scaffold` is `requires:`'d, so `anu-rebuild wave-execute` can call it.
3. **Should `Inputs/Salvaged/` be a hard rule or a recommended convention?** Recommend hard rule: the auditability benefit is too high to leave to convention.
4. **Per-wave commit cadence**: required or recommended? Recommend required; `anu-doctor project` checks for "no work since last cohort closeout commit" as a P11 (new) check.
5. **What's the minimum series count below which `anu-rebuild` shouldn't be invoked?** Currently set to 20; could be configurable. Lean keep as guideline with override.
