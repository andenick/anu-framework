## Purpose

`anu-rebuild` wraps the salvage-and-port workflow for taking an existing data construction project (typically a predecessor like ST2 or an academic replication that lives outside the Anu Framework) and producing a fresh, Anu-Framework-native implementation.

It is the meta-skill that **orchestrates the other 19 skills through a 6-wave + closeout cadence** proven to take a 60-series replication from blank scaffold to 100% PASS distribution-ready in ~12-15 focused sessions (RMWND baseline). With the friction-point remediations in v11.0 absorbed, the target is ~7-10 sessions.

The skill is general — works on any large data-construction predecessor regardless of domain. The RMWND rebuild was economic data; the same workflow applies to capital markets, sectoral employment, environmental accounts, or any other domain where you have a predecessor + source documents + the intention to re-author under audit-grade discipline.

---

## When to invoke

Invoke `anu-rebuild` when **all four** conditions hold:

1. **A predecessor project exists** with valuable data, methodology, and/or correspondence that should be salvaged rather than recreated from raw sources.
2. **The target is a new public-facing repository**, fully Anu-Framework-native, ready for `/anu-publish` distribution.
3. **Series count is large enough** (>= 20) that ad-hoc rewriting would lose discipline. Below 20 series, use `anu-pipeline` directly.
4. **The predecessor is NOT already Anu-Framework-native at the current version.** If it is, `anu-ingestion migrate-scheme` alone suffices.

## Six-wave workflow

Each wave is a discrete sub-command. The workflow is sequential but waves can sub-batch (sprints).

### Wave 0 — Foundation

**Goal**: empty scaffold + curated salvage + registry skeleton + crosswalk + framework audit clean.

```bash
anu-rebuild salvage <predecessor_root> --to Inputs/Salvaged/ [--manifest manifest.csv]
anu-rebuild crosswalk --predecessor Inputs/<pred>/series_registry.json \
                      --prefix-scheme '{"primary":"S","external":"ES","analytical":"AS"}' \
                      --output MIGRATION/crosswalk.csv
anu-rebuild scaffold --target . --crosswalk MIGRATION/crosswalk.csv
anu-doctor                        # framework consistency
```

| Sub-command | What it produces |
|---|---|
| `salvage` | `Inputs/Salvaged/` with predecessor's irreplaceable artifacts. Read-only afterwards. |
| `crosswalk` | `MIGRATION/crosswalk.csv` with `old_id,new_id,name,chapter_or_source,year_range,notes`. |
| `scaffold` | Full `Technical/` tree: registry skeleton, empty pipeline phases, `.gitignore`, `.publish_ignore`, data/code/docs subdirs. Initialized git repo with initial commit. |

**Acceptance**: `Inputs/Salvaged/` populated; `Inputs/<predecessor>/` untouched; crosswalk reviewed/approved; registry has `prefix_scheme` + series stubs; `anu-doctor` clean; first commit.

### Wave 1..N — Per-cohort implementation

Each cohort (chapter, wave, study group) runs:

```bash
anu-rebuild wave-execute --cohort <name> [--strict]
```

Internally orchestrates:

1. `anu-research mine-chapter <N>` OR `anu-research port --from <predecessor>/research/ --id-map MIGRATION/crosswalk.csv`
2. `anu-adequacy <N>` — must score >=80 ADEQUATE (gate)
3. `anu-ingestion batch-create-dpr --cohort <name> --template default`
4. `anu-extension batch-create-epr --cohort <name>` (for extendable subset)
5. `anu-scaffold generate --cohort <name>` — emits L01/P02/V03 stubs
6. **Agent fills scaffold stubs** with construction-specific logic
7. `python run.py --series <each in cohort> --validate-only` until all PASS
8. `anu-chopped --cohort <name>` + `anu-extenbook --cohort <name>`
9. `anu-review <chapter>` — must score >=85% with D13 (data authenticity) and D14 (outward intelligibility) GREEN
10. `git commit -m "Wave <N>: <cohort> closeout - N series PASS"`

**Acceptance per cohort**: every series has status `validated:<scope>`; `anu-review` >=85% with D13 GREEN; all artifact types present; cohort closeout commit; `anu-doctor project` CLEAN for the cohort scope.

### Wave N+1 — Distribution

```bash
anu-publish audit                              # iterate until CLEAN (zero internal refs)
anu-publish generate --profile data+pipeline   # public GitHub repo tree
anu-drive generate                              # Google Drive consumer bundle
anu-archive generate                            # audit-grade .zip
anu-review --whole-project                     # D13 + D14 GREEN required
git remote add origin <public_repo_url>
git push -u origin main
gh release create v1.0 Outputs/Archive/*.zip
```

**Acceptance**: `anu-publish audit` CLEAN; `Outputs/Publish/` valid GitHub-ready tree; `Outputs/Drive/` opens without code; `Outputs/Archive/...zip` extracts and SHA-256 verifies; fresh-clone smoke test passes in <5s; `anu-review` whole-project >=85% with D14 GREEN; GitHub Release v1.0 published.

### Wave N+2 — Polish

```bash
anu-doctor project                              # 10 P##-checks (DPR coverage, L/P/V triad, etc.)
/sphdarp <chapter_pages>                        # KB enrichment if D1 < 80 anywhere
```

If `anu-doctor project` flags issues, fix and re-run before declaring done. Log any framework gaps the rebuild surfaced in `MIGRATION/skill_updates_RFC.md` for next-cycle framework PRs.

## Cross-cutting policies

### `Inputs/Salvaged/` is read-only

Once Wave 0's `salvage` completes, nothing else writes to `Inputs/Salvaged/`. Pipeline code does not reference the predecessor's tree directly. All curated copies live in `Inputs/Salvaged/`. Salvage list is auditable.

### Crosswalk maintained for one release cycle

`MIGRATION/crosswalk.csv` is the canonical mapping. Referenced during research-JSON port, ID-scheme migration, and consistency checks. After v1.0 it freezes as historical doc.

### Divergences logged centrally

Every divergence from the predecessor — ingestion, extension, manual adjustment, scaffolding, scrub — is logged via `register_divergence()` (in the framework's shared helper) into top-level `DIVERGENCE_REGISTER.json`. No silent rewrites.

### Status field discipline

Every registry entry declares a status from the standardized enum (per `anu-ingestion` v4.1 schema). Pending series carry explicit dependency tokens (`pending:K_star`). Partial coverage carries explicit reason (`partial:14_year_coverage`). `anu-doctor project` validates.

### No synthetic data, ever

`anu-ingestion`'s Source Authenticity Verification applies universally. Pending series carry explicit `NaN` with provenance markers, not interpolated guesses. The `anu-review` D13 gate enforces.

### Per-wave closeout commits

Each wave ends with a commit listing every series gained, every artifact produced, validation status. Git log is the project progress report. No silent merges.

## Anti-patterns

### Do not copy predecessor outputs as starting point

The predecessor's `data/final/*.csv` are reference benchmarks at most. Re-derive every value from documented sources. Copying outputs creates silent dependencies; predecessor bugs ship to you.

**Real RMWND example**: ST2's `T507.csv` carried a NIPA-proxy surplus ratio of 0.5698 at 1948. The book's identity S/Y = e/(1+e) with e=1.70 yields 0.6296. RMWND's S507 was re-derived from S505 and S504; the divergence was caught and documented. Had I copied ST2's T507.csv, the bug would have shipped.

### Do not skip per-wave review

The `anu-review` gate is the framework's quality enforcement. Skipping pushes defects downstream where they cost 10x more to fix. The 85% gate is a hard line.

### Do not fabricate "pending" data

Series that can't be implemented yet carry explicit `NaN` and `status: pending:<dependency>`. NEVER fill pending years with synthetic averages, linear interpolations, or "reasonable estimates."

### Do not let the crosswalk drift

After Wave 0 approval, every artifact uses new IDs consistently. The crosswalk itself + salvaged read-only artifacts in `Inputs/Salvaged/` + `MIGRATION/_port_*.py` one-off porters are the only places predecessor IDs may appear.

### Do not write generators outside `MIGRATION/`

One-off scripts that don't belong in the public release live in `MIGRATION/`. The directory is in `.publish_ignore` by default. Keeps the public tree clean.

## When NOT to use

| Scenario | Use instead |
|---|---|
| No predecessor; fresh project | `anu-pipeline init` directly |
| Predecessor already at current Anu Framework version | `anu-ingestion migrate-scheme` only |
| Series count < 20 | Per-series invocation of existing skills |
| Predecessor data is unsalvageable | Treat as fresh project with extra digitization step |
| Single-chapter rebuild within larger Anu-native project | `anu-pipeline --from-stage 3` for that chapter |

## Generalization beyond economic data

The RMWND build was economic data (Shaikh-Tonak Marxian aggregates). The workflow generalizes to any domain meeting the four invocation conditions.

| Domain | Predecessor | Salvage candidates | Per-cohort partition |
|---|---|---|---|
| Capital markets | ETL pipeline + analyst spreadsheets | Cleaned tick data, methodology, correspondence | by asset class |
| Sectoral employment | BLS CES historical notebook | Bridge tables, classification mappings | by NAICS sector |
| Environmental accounts | EPA + Eurostat compiled datasets | Concordance tables, satellite-data docs | by pollutant / by country |
| Heterodox replication | Author's R or Stata code | Author correspondence, classification spreadsheets | by chapter / by paper |

In every case the 6-wave cadence applies. Only per-cohort partition varies. Salvage targets, crosswalk semantics, and scrub patterns are domain-agnostic.

## Integration with Anu Framework

| Skill | Relationship |
|---|---|
| `anu-doctor` (required) | Runs at end of Wave 0 (framework audit) and Wave N+2 (project audit) |
| `anu-ingestion` (required) | Wave 0 `migrate-scheme`; every per-cohort wave uses `batch-create-dpr` |
| `anu-publish` (required) | Wave N+1 distribution |
| `anu-pipeline` (required) | `anu-rebuild` sits above `anu-pipeline`, prescribing project-scope cadence |
| `anu-scaffold` (required) | Per-cohort wave step #5 |
| `anu-research` | Per-cohort wave step #1 (port or fresh mine) |
| `anu-adequacy` | Per-cohort wave step #2 (gate) |
| `anu-extension` | Per-cohort wave step #4 (extendable subset) |
| `anu-chopped`, `anu-extenbook` | Per-cohort wave step #8 |
| `anu-review` | Per-cohort wave step #9; Wave N+1 whole-project audit |
| `anu-drive`, `anu-archive` | Wave N+1 distribution |

`anu-rebuild` does NOT replace `anu-pipeline` — it overlays a project-scoped workflow on top of `anu-pipeline`'s 8-stage skill orchestration.

## Verification

The skill is correctly implemented and useful when:

1. **A new rebuild target measured against RMWND baseline shows >=30% session-count reduction** (12-15 → 7-10 sessions).
2. **The 6-wave cadence produces complete artifacts without ad-hoc generator scripts.** Project-level `MIGRATION/_gen_*.py` files become unnecessary.
3. **Every divergence from the predecessor is logged in `DIVERGENCE_REGISTER.json`** — not in ad-hoc markdown.
4. **`anu-doctor project` is CLEAN at Wave N+2.**
5. **The public release passes the scrub audit and external review** without manual scrubbing iterations.
6. **The empirical findings of the predecessor (the book, the paper) are reproduced** post-rebuild.
